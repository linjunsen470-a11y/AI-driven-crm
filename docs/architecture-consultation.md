# Maoshan AI CRM - 架构设计咨询文档

> **文档用途**: 向 AI Agent 咨询架构可行性、技术选型和竞品分析  
> **项目阶段**: MVP 开发中，即将部署上线  
> **咨询重点**: 分离部署架构合理性、技术栈选型、同类产品借鉴

---

## 1. 项目概述

### 1.1 业务背景

面向**高端养老社区/康养机构**的 AI 驱动的销售 CRM 系统。

**目标用户**: 康养社区销售团队（10-50 人规模）  
**核心场景**: 
- 销售人员通过电话/微信/面谈与客户（老人及其子女）沟通
- 需要记录和追踪复杂的客户需求（健康状况、预算、决策链、入住时间等）
- AI 辅助提取客户画像、生成跟进建议、处理录音和截图

### 1.2 核心功能模块

| 模块 | 功能描述 | 状态 |
|------|----------|------|
| **客户管理** | 客户档案、标签体系、决策链分析 | ✅ 已实现 |
| **互动记录** | 通话录音、微信截图、面谈笔记的录入与 AI 处理 | ✅ 已实现 |
| **AI 处理** | 语音识别、OCR、客户画像提取、跟进建议生成 | 🔄 待集成 n8n |
| **确认工作流** | AI 提取的数据需人工确认后才写入主数据 | ✅ 已实现 |
| **任务提醒** | 基于规则/AI 的智能跟进提醒 | 📋 设计中 |
| **知识库** | 产品资料、FAQ、销售话术支持 | 📋 设计中 |
| **AI 对话** | 销售助手，基于知识库回答销售问题 | 📋 设计中 |

### 2.2 分离部署决策理由

| 组件 | 部署位置 | 理由 |
|------|----------|------|
| Next.js 应用 | VPS 1 (新加坡) | 靠近用户（中国），低延迟访问 |
| PostgreSQL | Supabase (托管) | 免运维，自动备份，连接池管理 |
| 文件存储 | MinIO (独立服务器) | 大文件存储，S3 兼容，成本可控 |
| AI 工作流 | n8n (VPS1 或独立) | 异步处理，与主应用解耦 |
| AI 模型 | OpenAI API | 无需自建 GPU 服务器 |

## 4. 数据库设计概要

### 4.1 核心实体关系

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│   Customer   │────◄│ Interaction  │────◄│    File      │
│   (客户)      │     │   (互动记录)  │     │   (文件)      │
└──────────────┘     └──────────────┘     └──────────────┘
       │                    │
       │              ┌─────┴─────┐
       ▼              ▼           ▼
┌──────────────┐ ┌──────────┐ ┌──────────┐
│    Task      │ │  AiJob   │ │ Customer │
│   (任务)      │ │ (AI任务)  │ │ Profile  │
└──────────────┘ └──────────┘ │Snapshot  │
                              └──────────┘
```

### 4.2 关键字段设计

**客户表 (Customer)** 包含完整的康养行业画像字段：
- 基础信息：姓名、电话、年龄、城市
- 退休信息：退休状态、职业类别、养老金范围
- 家庭信息：婚姻状况、居住安排、子女支持度
- 决策信息：决策人、意向评分、预算范围、决策阶段
- 健康信息：自理能力、护理需求、健康风险
- 入住意向：参观意愿、试住意愿、预计入住时间

**互动表 (Interaction)** 支持多模态内容：
- 联系方式：电话、微信、面谈
- 内容类型：音频、图片、文本
- AI 处理状态：待处理、处理中、已完成、失败
- 确认工作流：AI 提取数据、人工确认状态

## 6. 待解决的关键问题

### 6.1 技术实现问题

1. **存储适配器**: 当前 `lib/storage/index.ts` 只实现了 `local` 驱动，`minio` 驱动需要基于 AWS SDK 实现 S3 兼容接口。是否有推荐的封装库或最佳实践？

2. **n8n 集成模式**: 
   - 方案 A: n8n 与 Next.js 同机部署（简单）
   - 方案 B: n8n 独立部署，通过 webhook 回调（解耦）
   - 对于 AI 工作流编排，哪种模式更适合生产环境？

3. **AI 工作流设计**: 
   - 语音文件上传 → n8n 触发 → Whisper 转录 → GPT 提取结构化数据 → 回调写入临时表 → 人工确认 → 合并到主表
   - 这个流程是否合理？是否有更优的异步处理模式？

4. **确认工作流的数据流**: AI 提取的数据需要先展示给销售确认，确认后才写入主数据。当前设计是在 `Interaction` 表存储 `aiExtractedData` 和 `confirmationStatus`，确认后合并到 `Customer`。这种模式是否清晰可维护？

### 6.2 架构设计问题

5. **分离部署的网络安全**: VPS1 (Next.js) 通过公网连接 Supabase 和 MinIO。除了 SSL 和强密码，是否还需要额外的安全措施（如 IP 白名单、VPC peering）？

6. **文件存储策略**: 录音文件较大（可能 10-50MB），MinIO 部署在独立服务器。是否需要 CDN 加速？还是直接通过预签名 URL 访问？

7. **数据库连接池**: Supabase 有连接数限制。Next.js Serverless 环境下 Prisma 的连接管理是否有注意事项？

### 6.3 产品功能问题

8. **竞品对标**: 市场上是否有类似 "AI 驱动的垂直行业 CRM" 产品可以参考？特别是：
   - 康养/养老行业 CRM
   - 支持 AI 客户画像提取的 CRM
   - 有确认工作流机制的 CRM

9. **AI 提取的准确性**: 康养客户画像字段很多（30+ 字段），一次性提取准确性可能不高。是否应该：
   - 分批提取（基础信息 → 家庭信息 → 决策信息）？
   - 还是在确认环节支持部分确认/修改？

10. **知识库 RAG**: 计划使用 OpenAI Embedding + 向量检索。对于产品资料这种结构化文档，是否有比纯向量检索更好的方案（如混合检索）？

### 8.2 关键技术决策记录

| 决策 | 选项 | 选择 | 理由 |
|------|------|------|------|
| 数据库 | Supabase vs 自建 PostgreSQL | Supabase | 免运维，自动备份，新加坡节点 |
| 存储 | MinIO vs 阿里云 OSS | MinIO | 成本可控，S3 兼容，已有域名配置 |
| AI 工作流 | n8n vs Dify vs 自研 | n8n | 可视化编排， webhook 集成成熟 |
| 部署架构 | 单机 vs 分离 | 分离 | 灵活扩展，职责分离 |
| 认证 | Better Auth vs NextAuth | Better Auth | 支持多数据库，类型安全 |

### 8.3 性能预期

| 指标 | 目标 | 说明 |
|------|------|------|
| 页面首屏 | < 2s | Next.js SSR + 静态优化 |
| API 响应 | < 500ms | 数据库查询优化 |
| 语音处理 | < 30s | 5分钟录音的完整处理链路 |
| 并发用户 | 50+ | 销售团队规模 |
