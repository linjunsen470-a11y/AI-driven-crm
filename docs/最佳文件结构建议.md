# 最佳文件结构建议

这份文档基于你当前仓库的实际情况来判断：

- 根目录有 `package.json`、`prisma/`
- 同时又有一个独立的 `next-app/`
- `next-app/` 内还有自己的 `.git`、`package.json`、`node_modules`

我的结论很明确：

`当前结构不是最佳实践。`

它现在更像是：

- 根目录在做“规划与数据库容器”
- `next-app/` 在做“真正的前端项目”

这种双根结构在项目早期还能忍，但一旦你继续接入：

- Prisma migration
- n8n webhook
- AI Prompt
- 知识库
- 文件上传
- Docker Compose

很快就会开始混乱。

---

## 1. 当前结构的主要问题

## 1.1 有两个“项目根”

当前你同时有：

- 根目录 `package.json`
- [next-app/package.json](/mnt/d/Maoshan/next-app/package.json)

这会带来几个问题：

- 依赖到底装在根目录还是 `next-app/` 不清楚
- Prisma 应该从哪里执行不清楚
- 脚本入口不统一
- CI/CD、Docker、部署时容易混乱

---

## 1.2 有两个 `node_modules`

当前存在：

- `/mnt/d/Maoshan/node_modules`
- `/mnt/d/Maoshan/next-app/node_modules`

这会导致：

- 版本漂移
- 体积膨胀
- 命令执行路径不稳定
- 排查依赖问题更困难

---

## 1.3 `next-app/` 里还有独立 `.git`

当前目录下有：

- [next-app/.git](/mnt/d/Maoshan/next-app/.git)

这意味着你的前端实际上像是一个嵌套仓库。

对于现在这个项目阶段，这通常不是你想要的结构，因为：

- 文档、Prisma、前端代码并没有在一个统一版本历史里
- 以后你接 `infra/`、`docker-compose.yml`、`n8n` 配置会更 awkward

---

## 1.4 文档、产品资料、提示词和运行时代码混在根目录

根目录里现在同时放了：

- 产品需求文档
- 背景知识库
- Prompt 模板
- 架构分析
- Prisma
- 依赖

这在探索期没问题，但进入开发期后，建议分层：

- `docs/` 放分析和规划
- `prisma/` 放数据库
- `apps/web/` 或项目根放 Next.js 应用
- `prompts/` 放运行时 Prompt
- `infra/` 放部署和本地基础设施

---

## 2. 最适合你现在阶段的结构

我不建议你现在上来就做 monorepo 全家桶。  
对你当前规模，最稳的做法是：

`单仓库 + 单主应用 + 清晰分层`

也就是说：

- 整个仓库只有一个真正的项目根
- Next.js 应用和 Prisma 在同一个根下
- 文档、Prompt、基础设施文件分目录管理

## 4. 这套结构为什么适合你的 AI CRM

## 4.1 `app/` 只放路由，不放太多业务逻辑

`app/` 适合放：

- 页面入口
- route layout
- route handler

不适合把所有业务代码都堆进去。

对于你这种系统，后面会有很多复杂逻辑：

- 客户画像合并
- AI 结果确认
- 批量导入录音
- 提醒规则
- 知识库检索

这些放进 `features/` 更清晰。

---

## 4.2 `features/` 用领域拆分，而不是技术拆分

这个项目最适合按业务领域拆：

- `customers`
- `interactions`
- `knowledge`
- `chat`
- `reminders`
- `ingestion`

原因是你后续最常见的修改方式，不是“我要改一个 hooks 文件”，而是：

- 我要改客户详情页和客户 schema
- 我要改录音导入和 interaction 提取
- 我要改提醒规则和提醒列表

领域化结构更适合长期演进。

---

## 4.3 `components/` 只放可复用 UI

建议划分：

- `components/ui/`：shadcn 原子组件
- `components/layout/`：侧边栏、顶栏、页面壳子
- `components/shared/`：通用业务组件
- `components/forms/`：通用表单部件
- `components/providers/`：Theme、Query、Auth Provider

不要把某个客户详情页专用组件也都扔到根 `components/`。

专属组件应该尽量靠近 `features/`。

---

## 4.4 `lib/` 只放真正跨领域的基础能力

适合放：

- Prisma client
- 环境变量校验
- logger
- 文件存储客户端
- AI provider 适配
- auth helper
- 通用工具函数

不适合放：

- 客户画像合并规则
- interaction 解析业务逻辑

这些更应该归到 `features/`。

---

## 4.5 `prompts/` 单独拆出来很有必要

你这个项目不是普通后台系统，Prompt 是核心资产之一。

建议不要把 Prompt 长期散落在：

- 某个 route handler
- 某个 service 文件
- 根目录随意一个 markdown

更好的做法是：

```text
prompts/
├─ extraction/
│  ├─ customer-profile.md
│  └─ call-ingestion.md
├─ sales-chat/
│  └─ customer-advice.md
└─ reminders/
   └─ followup-reasoning.md
```

这样你以后做版本管理、A/B 测试和提示词升级会轻松很多。

---

## 4.6 `docs/` 应该收拢你现在的 Markdown

你现在根目录的文档已经很多了，建议后续收拢成：

```text
docs/
├─ product/
│  ├─ Product Requirements Document.md
│  ├─ background.md
│  └─ AI-CRM knowledge base.md
├─ architecture/
│  ├─ AI-Driven-CRM-架构分析与开发建议.md
│  ├─ advice.md
│  └─ 最佳文件结构建议.md
├─ prompts/
│  ├─ design.md
│  └─ AI-信息提取Prompt模板.md
└─ implementation/
   ├─ MVP-API-与工作流设计.md
   └─ 实施步骤、依赖与模板建议.md
```

这会让仓库根目录干净很多。

## 5. 你现在最该修的结构问题

优先级从高到低：

### 1. 去掉双根结构

要么：

- 把 `next-app/` 内容提升到仓库根目录

要么：

- 把整个仓库改成正式 monorepo，并把 `next-app/` 放到 `apps/web/`

对你当前阶段，我更推荐第一种：

`把 next-app 变成唯一主项目根`

### 2. 删除嵌套 `.git`

[next-app/.git](/mnt/d/Maoshan/next-app/.git) 不建议保留。

### 3. 合并依赖管理

最终只保留一个：

- `package.json`
- `pnpm-lock.yaml`
- `node_modules`

### 4. 文档归档到 `docs/`

不要让根目录长期堆满规划文档。

### 5. 把 Prompt 从“分析文档”里分离出来

运行时 Prompt 和产品说明文档最好分开。

## 6. 单应用结构 vs Monorepo

## 6.1 我现在推荐你用单应用结构

也就是：

- 仓库根就是 Next.js 应用根
- `prisma/`、`docs/`、`infra/` 都在根目录旁边

适用原因：

- 你现在只有一个真正的 Web 前端
- n8n 是外部工作流，不需要单独写成包
- 还没有多个独立 Node 服务

这是现在最省心、最符合最佳实践的方案。

---

## 6.2 什么时候再升级成 Monorepo

当你出现下面任一情况时，再考虑：

- 有独立 worker 服务
- 有独立共享 UI 包
- 有独立 SDK 包
- 有独立管理端和客户端

到那时再演进成：

```text
apps/
  web/
packages/
  ui/
  config/
  prompts/
```

会更合理。

但现在没必要过早复杂化。

## 7. 我对你当前仓库的最佳实践结论

### 最佳实践结论

对于你现在的阶段，最好的结构不是：

- 根目录放规划 + 嵌套一个 `next-app/`

而是：

- `Next.js 应用成为唯一项目根`
- `docs/`、`prisma/`、`prompts/`、`infra/` 作为清晰分层目录
- `features/` 按业务领域拆分

---

## 8. 推荐迁移顺序

如果你后面要正式整理结构，我建议按这个顺序迁移：

1. 确定“唯一项目根”是仓库根
2. 把 `next-app/` 内的应用文件迁移到根目录
3. 合并 `package.json` 与 lockfile
4. 删除 `next-app/.git`
5. 删除多余 `node_modules`
6. 新建 `docs/`
7. 新建 `prompts/`
8. 新建 `features/`
9. 把当前分析文档归档
10. 再开始正式开发页面和接口

---

## 9. 一句话建议

如果你现在就要整理结构，我建议直接把：

`next-app/` 提升为唯一主应用根，并把文档归档到 `docs/`，再按 `features + lib + prompts + prisma + infra` 的结构继续开发。
